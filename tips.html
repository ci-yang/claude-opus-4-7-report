01 Auto Mode

不用再守著 permission prompt

Opus 4.7 很適合跑長任務——deep research、重構、iterate 到命中 perf benchmark。Auto mode 把 permission 請求送 classifier 判斷，安全就自動批准。

從 babysit 到放手跑 以前要嘛你守在旁邊、要嘛 --dangerously-skip-permissions 硬上。Auto mode 把 permission 請求送 classifier，安全就自動批准。
同時跑多個 Claude 一個在 cook，你切到下一個。不用等它跑完才能做下一件事——開發節奏整個被放大。
支援方案 Max / Teams / Enterprise 用戶可用；目前僅支援 Opus 4.7 模型。 
$ claude
# CLI 按 Shift + Tab 進入 Auto Mode
→ permission prompt 送 classifier 判斷
→ 安全指令自動批准，不用守在旁邊
→ 多個 session 並行跑，切換自如
02 /fewer-permission-prompts

掃描歷史，自動建議 allowlist

新的 skill，會掃你 session 歷史，找出那些「明明安全卻一直被問」的 bash 與 MCP 指令，直接幫你列出該加進 allowlist 的清單。

掃 session 歷史 找那些你重複批准的指令——git status、npm test、curl <api>、ls -la。
產出 allowlist 清單 直接給你一份建議清單，你審過後加進 .claude/settings.json，往後就不再被問。
何時用 不想開 Auto Mode，但被 permission prompt 打斷到煩的時候。
延伸：Claude Code 也有個 /less-permission-prompts 是社群的早期版，官方新 skill 名稱為 /fewer-permission-prompts，邏輯類似但 Boris 建議用官方版。
03 Recaps

長任務的續航摘要 · 回來一目了然

這週稍早為了 Opus 4.7 發布的小功能。Claude 跑到一半你跳開幾分鐘、幾小時再回來，Recaps 會直接告訴你 agent 剛才做了什麼、接下來要做什麼。

剛才做了什麼

agent 已完成的工作一覽——不用翻 log 往前倒帶。

接下來要做什麼

下一步 plan，讓你能直接決定「繼續跑」或「調整方向」。

適用場景

任何長跑任務——deep research、大型重構、跨多檔案的 feature。

04 Focus Mode

/focus · 只看最終結果

CLI 新功能——把中間所有過程都隱藏，只顯示最後結果。4.7 已經到一個可以被信任的水準，Boris 自己大部分任務都只看結果。

過程全部收折 bash 呼叫、tool 輸出、中間 plan——都摺起來，只留最後結果給你看。
前提：信任模型 要用得起來，你得先到達「信任模型跑對 commands、改對地方」的水準。4.7 幫很多任務跨過這個門檻。
切換方式 /focus 指令 on/off——不用的時候切回一般模式，看完整過程。
Boris 原話：「4.7 已經可以信任它跑對指令、改對地方了——我只看最終結果就好。」
05 /effort

Effort 等級怎麼選 · xhigh 是甜蜜點

4.7 改用 adaptive thinking，不再用 thinking budget。要調它想多久？用 effort。Boris 自己大部分任務用 xhigh，最難的任務才上 max。

低 effort 走「快、省 token」路線，高 effort 走「最聰明」路線。用 /effort 指令切換；除了 max 之外都會記住。

06 Verification · 最重要的一個

給 Claude 一個驗證方式 · 輸出品質 2–3×

Boris 說這招過去就能讓 Claude 輸出品質 2–3×，4.7 之後更關鍵——因為模型跑更久、更 agentic，沒驗證你根本不知道它是不是在幻覺。

Backend

教 Claude 怎麼起你的 server / service，讓它自己跑 end-to-end 測試，看 response 才算完成。

Frontend

用 Claude Chromium extension 控 browser，讓它真的點點看、看到 UI 才算完成。

Desktop / 其他

用 computer use，讓它實際操作介面完成任務，不是只改 code。

Boris 的 /go skill

他常用 prompt 長這樣：「Claude do X /go」。/go 是個 skill，會做三件事：

1
跑 end-to-end 測試用 bash / browser / computer use 確認真的能動
2
跑 /simplify把冗餘 wrapper / 過度工程的部分刪乾淨
3
發 PR測過、簡化過、才進 review
關鍵：長任務尤其需要——你跳開回來後，才會知道 code 是真的能動，不是 Claude 幻覺出一份看起來很漂亮但跑不起來的成果。
✓ Closing

真正的躍進 · 要花時間調整 workflow

4.7 的體感，不在於模型變聰明了多少，而在於你願不願意讓它跑得更久、管得更少。

Boris 收尾原話：「4.7 是 significant step up，但你要花時間調整 workflow——讓它跑更久、更 agentic。用舊 workflow 你只會感覺它有進步；真正的躍進，要等你改變習慣後才會到來。」
